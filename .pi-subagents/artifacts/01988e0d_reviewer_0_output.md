## Review
- **Correct:** Two-layer content is retained during ordinary transitions; source layers become disabled but remain visible through opacity bindings (`Bar.qml:340-455`).
- **Correct:** Equal-height modes receive zero vertical offset (`Bar.qml:255-257`), and morph animations use non-bouncing cubic easing over 280 ms (`Bar.qml:290-312`).
- **Correct:** Transition metadata records source/target monitors, and local mapping falls back to workspaces on unaffected monitors (`StatusData.qml:497-515`, `Bar.qml:224-227`, `Bar.qml:317-324`).
- **Correct:** Launcher query is preserved when moving an already-visible launcher between monitors (`StatusData.qml:525-540`).
- **Blocker:** Rapid/repeated transitions snap opacity instead of continuing from the rendered state. Metadata records only the current logical mode (`StatusData.qml:497-515`), while every serial change stops the running animation and resets progress to either `0` or `1` (`Bar.qml:270-278`). For example, a second volume-wheel event during the initial 280 ms produces `volume → volume`; this immediately sets the volume layer fully opaque and removes the still-visible workspace source. Switching to another mode mid-transition similarly promotes a partly entered logical source to opacity 1 and drops the previous rendered source instantly. This violates the rapid-transition and coexistence requirements. Same-mode retriggers should leave the running content animation untouched; mode changes during an active transaction need continuity from current rendered opacity/offset values.
- **Note:** No runtime visual validation or QML linting was performed. Dynamic monitor removal during an active transition remains a residual risk.
