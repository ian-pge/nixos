# Research: Hyprland v0.55.4 `specialWorkspace` `slidevert` vs. Qt Quick client-side animation

## Summary

Hyprland v0.55.4 implements this effect by animating a workspace-owned `Vector2D` render offset (and, for every special-workspace open/close, workspace alpha), sampling its configured curve from elapsed time, damaging the affected workspace, and requesting an animation frame from the target output. At render time the compositor adds that offset directly to every unpinned window's position, so all workspace textures, decorations, popups, clipping, dimming, and blur participate in one compositor frame.

Qt Quick can closely reproduce the *visual trajectory of Quickshell's own surface content*—especially with `YAnimator`/`XAnimator`, the same duration, and a matching Bézier—but it cannot reproduce the compositor's atomic movement of arbitrary workspace clients or obtain the compositor's internal animation progress through Quickshell's Hyprland IPC API. Native motion can therefore have better frame pacing and coherence, although a light, single-window Qt Quick scene using the threaded render loop can itself be very smooth.

## Findings

1. **[informational] Exact v0.55.4 implementation and endpoints** — Tag `v0.55.4` resolves to commit `a0136d8c04687bb36eb8a28eb9d1ff92aea99704`. `CDesktopAnimationManager::startAnimation(PHLWORKSPACE, ...)` selects `specialWorkspaceIn` or `specialWorkspaceOut` for both `m_alpha` and `m_renderOffset`. `slidevert` sets the incoming offset to `y = ±(monitor height + general:gaps_workspaces) × percentage`, then targets `(0,0)`; outgoing uses the opposite signed endpoint. Direction can be overridden with `top`/`bottom`. A special workspace additionally always animates alpha `0→1` on open and `1→0` on close, even for plain `slidevert` rather than `slidefadevert`. [DesktopAnimationManager.cpp, lines 243–372](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/managers/animation/DesktopAnimationManager.cpp#L243-L372) [tag ref](https://api.github.com/repos/hyprwm/Hyprland/git/ref/tags/v0.55.4)

2. **[informational] Workspace variables are compositor animation objects with whole-workspace damage** — On workspace initialization, Hyprland creates animated variables for `m_renderOffset` and `m_alpha`, associates both with the workspace, and gives both `AVARDAMAGE_ENTIRE`. Thus animation is not client geometry negotiation; it is a compositor-side transform over already available client buffers. [Workspace.cpp, lines 34–42](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/desktop/Workspace.cpp#L34-L42)

3. **[informational] Timing is elapsed-time curve sampling, not a fixed one-update-per-display-frame loop** — `handleUpdate()` calls `getCurveStep()`, interpolates from `begun()` to `goal()`, then invokes the variable update callback. `frameTick()` is driven by an event-loop timer, guards actual ticks to at least 1 ms apart, and reschedules a 1 ms timer while animation remains active. In `tick()`, Hyprland damages the old state, updates all animated variables, damages the new state, and asks the compositor to schedule the monitor with `AQ_SCHEDULE_ANIMATION`. The configured `SPEED` value is actually duration in deciseconds (`1 = 100 ms`), so the Qt equivalent duration is `speed × 100 ms`; the visible samples are ultimately quantized by output presentation. [AnimationManager.cpp, interpolation lines 130–153](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/managers/animation/AnimationManager.cpp#L130-L153) [tick/damage/schedule lines 155–312](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/managers/animation/AnimationManager.cpp#L155-L312) [timer scheduling lines 314–350](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/managers/animation/AnimationManager.cpp#L314-L350) [Hyprland animation syntax/timing](https://wiki.hypr.land/Configuring/Advanced-and-Cool/Animations/)

4. **[informational] The offset is consumed at final composition** — In `IHyprRenderer::renderWindow`, `REALPOS` is the window's animated real position plus the workspace render offset for every non-pinned window. Workspace alpha is also multiplied into final surface alpha. Floating-window clipping is translated by the same workspace offset. This is why the entire special workspace looks like one rigid, coherent object rather than independent clients reacting to IPC. [Renderer.cpp, lines 539–606](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/render/Renderer.cpp#L539-L606) [floating clipping, lines 623–627](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/render/Renderer.cpp#L623-L627)

5. **[informational] Output scheduling remains compositor-owned** — The animation manager only requests an output animation frame after updating/damaging. `CMonitorFrameScheduler::onFrame()` receives the output frame opportunity and renders normally; with the newer explicit-sync scheduler it can also begin rendering early when a prior frame missed its refresh deadline. This puts animation state, damage, rendering, and output scheduling in the same process and lets Hyprland choose the state nearest the compositor's latch/present opportunity. [MonitorFrameScheduler.cpp, lines 82–132](https://github.com/hyprwm/Hyprland/blob/a0136d8c04687bb36eb8a28eb9d1ff92aea99704/src/helpers/MonitorFrameScheduler.cpp#L82-L132)

6. **[medium practical mismatch] Qt Quick adds a separate producer/render/commit stage** — In Qt's threaded render loop, a QML change requests a frame; Qt synchronizes QML state to the scene graph, renders on its render thread, and swaps/submits the client buffer. Hyprland must then receive and composite that buffer on a later output frame. A late GUI/render-thread update or Wayland commit can therefore miss the compositor's current latch, whereas Hyprland's native offset is selected inside the same compositor render cycle. This is a plausible reason native motion *may* look smoother; it is not proof that Qt always stutters. [Qt Quick threaded frame sequence](https://doc.qt.io/qt-6/qtquick-visualcanvas-scenegraph.html#threaded-render-loop-threaded)

7. **[medium practical mismatch] Qt animation pacing depends on render-loop topology** — Qt says ordinary `NumberAnimation` is timer-driven (typically 16 ms) in the basic loop. The threaded loop installs render-synchronized drivers for normal animations and render-thread `Animator` types, but it falls back to timers when more than one `QQuickWindow` is visible because there are multiple render-thread synchronization points. That is particularly relevant to Quickshell configurations creating a `PanelWindow` per monitor or multiple overlay windows. Verify with `qt.scenegraph.general`; do not assume all QML animation is vsync-driven. [Qt “Driving Animations” and multi-window fallback](https://doc.qt.io/qt-6/qtquick-visualcanvas-scenegraph.html#driving-animations)

8. **[low recommendation] Best Quickshell approximation** — Animate a child container inside a stable, transparent, monitor-sized `PanelWindow`, not the layer-shell window's anchors/margins/exclusive zone. Use `YAnimator` (or an item transform animator) where possible so progression can occur on Qt's render thread; Qt documents that Animator types can run there. Set `duration` to Hyprland speed × 100 ms, duplicate the configured cubic Bézier with a Qt easing curve, use a start/end distance equal to logical monitor height plus the intended gap, and animate opacity concurrently because special workspaces fade even under plain `slidevert`. `FrameAnimation` is preferable to a 16 ms `Timer` for custom imperative progression. [XAnimator/Animator behavior](https://doc.qt.io/qt-6/qml-qtquick-xanimator.html) [FrameAnimation guidance](https://doc.qt.io/qt-6/qml-qtquick-frameanimation.html) [Quickshell PanelWindow semantics](https://quickshell.org/docs/master/types/Quickshell/PanelWindow/)

9. **[high capability boundary] What Quickshell cannot replicate** — Public Quickshell Hyprland integration exposes workspace models and event-socket notifications, not `m_renderOffset`, curve progress, damage, output frame scheduling, or a transaction that moves third-party workspace surfaces. A Quickshell surface can animate its own Qt scene and can visually cover/snapshot an area if separately captured, but it cannot make Hyprland's actual app windows, popups, decorations, blur/dim state, hit testing, and workspace lifecycle follow that client animation atomically. IPC events can start an approximation, but event delivery and independent clocks prevent phase-perfect lockstep. [Quickshell Hyprland API](https://quickshell.org/docs/types/Quickshell.Hyprland/Hyprland/) [Quickshell WlrLayershell surface role](https://quickshell.org/docs/v0.3.0/types/Quickshell.Wayland/WlrLayershell/)

## Practical implications

- To match *feel*, copy duration, curve, direction, logical travel distance, and simultaneous opacity. Check whether the actual leaf is `specialWorkspaceIn` or `specialWorkspaceOut`; they may inherit or be overridden separately.
- Prefer one persistent surface per output and animate a lightweight child subtree. Recreating/showing a window at trigger time adds allocation, mapping, and first-buffer latency.
- Prefer `YAnimator`/render-thread animation for a simple translation. Use `FrameAnimation` only where the target changes dynamically; avoid a QML `Timer { interval: 16 }`.
- A full-screen transparent surface costs compositing bandwidth and may interact with blur. Configure its input mask and exclusion behavior so the invisible region neither blocks input nor reserves layout space.
- If the required result includes real application windows moving with correct input and blur throughout, keep the effect compositor-native. Quickshell is suitable only for its own overlay/content or a visual facsimile.
- Test frame pacing on the actual monitor topology. Multiple Quickshell windows, mixed refresh rates, GPU load, and the active Qt render loop can change results; enable `qt.scenegraph.general` and `qt.scenegraph.time.renderloop`.

## Gaps / residual risks

- The precise Bézier control points and `specialWorkspaceIn`/`Out` speed are user-configuration dependent; no configuration snippet was supplied, so no single millisecond duration or Qt easing tuple can be asserted.
- Hyprland's `getCurveStep()` implementation resides in the pinned `hyprutils` dependency, not this repository. The call path and documented duration units are verified, but this brief did not resolve the exact hyprutils commit and line-level elapsed-time formula.
- “Looks smoother” is workload-, driver-, refresh-, and topology-dependent. The architecture explains likely pacing differences but is not a benchmark; confirm with presentation timing or high-speed capture on the target setup.
- Quickshell and Qt versions were not specified. Availability of `FrameAnimation` requires Qt 6.4+, and actual render-loop selection must be observed at runtime.
